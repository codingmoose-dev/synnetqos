# Results

This directory contains the main tabular outputs generated by the SynNetQoS workflow. The results are organized by evidence type: generator reproducibility, dataset integrity, Monte Carlo stability, external alignment against selected real-world measurement datasets, and leakage-aware machine-learning benchmarking.

These outputs document the reproducibility, internal consistency, selected external alignment, and benchmark learnability of the generated SynNetQoS dataset. External-alignment results are interpreted as distributional sanity checks for selected radio and QoS/QoE variables, while the machine-learning outputs evaluate structured learnability under leakage-aware synthetic benchmark settings.

## Result groups

### `generator/`

This folder contains outputs related to the generated SynNetQoS dataset itself.

- `generator_config.json` records the main generator settings, random seed, and dataset SHA-256 hash. This file supports reproducibility by making the generated dataset traceable to a fixed configuration.
- `dataset_integrity_summary.csv` checks whether the expected number of rows was generated and whether timestamp ordering was preserved.
- `dataset_schema.csv` records column names, data types, and missing-value counts.

  `Video_Quality` and `Video_Quality_Label` are conditional QoE variables. They are generated only for rows where `App_Type` is `Streaming`; for Browse, Gaming, and Call sessions, these fields are intentionally left empty because video-quality scoring is not applicable. Therefore, the missing values reported for these two columns in `dataset_schema.csv` should be interpreted as non-applicability rather than incomplete data.

- `numerical_range_summary.csv` summarizes the observed numerical ranges of key synthetic variables.
- `outlier_report.csv` reports distributional extremes using an IQR-based rule. These values are diagnostic and should not automatically be treated as invalid.
- `monte_carlo_run_metrics.csv` stores run-level metrics from repeated generator runs.
- `monte_carlo_stability_summary.csv` summarizes the stability of selected generator outputs across repeated runs.

Together, these files support the claim that SynNetQoS is reproducible and that its generated variables can be audited systematically.

### `external_alignment/`

This folder contains comparisons between SynNetQoS and selected external measurement datasets from Vienna phone measurements and Campus QoS throughput data.

- `external_feature_mapping.csv` defines how synthetic variables were mapped to external measurement variables.
- `controlled_5g_subset_audit.csv` documents the filtering steps used to construct the controlled 5G SynNetQoS subset for Campus QoS comparison.
- `external_validation_vienna_phone_summary.csv` compares SynNetQoS against Vienna phone LTE/5G measurements for RSRP and throughput variables.
- `external_validation_campus_controlled_5g_summary.csv` compares the controlled SynNetQoS 5G subset against Campus QoS throughput and jitter measurements.
- `external_alignment_summary.csv` combines the main external-alignment summaries into one table.
- `external_validation_diagnostic_summary.csv` gives descriptive statistics for synthetic and external subsets.
- `external_alignment_interpretation_flags.csv` highlights variables where synthetic medians are noticeably higher or lower than the external references.

The strongest external-alignment evidence is observed for RSRP and selected download-throughput/jitter comparisons. Upload throughput is retained as secondary evidence because alignment is less consistent across the external references.

### `external_alignment/supplementary/`

This folder contains supporting diagnostics that are useful for transparency but should not be treated as main-paper evidence.

- `external_validation_vienna_positive_downlink_summary.csv` reports a positive-throughput-only Vienna comparison.
- `external_rsrp_to_throughput_trend.csv` summarizes the binned relationship between RSRP and download throughput.
- `external_spearman_rsrp_throughput.csv` reports source-specific Spearman correlations between RSRP and download throughput.
- `vienna_scanner_rsrp_summary.csv` summarizes Vienna scanner RSRP measurements. Scanner data is treated as RSRP-only supplementary evidence and is not mixed with connected-phone throughput validation.

These files are useful for auditability and supplementary discussion, but they should not be used to claim full real-world distribution matching.

### `ml_benchmark/`

This folder contains the leakage-aware machine-learning benchmark outputs for dropped-connection prediction tasks.

- `ml_task_definitions.csv` defines the three benchmark tasks:
  - Task A: same-timestep dropped-connection label recovery using full current QoS/QoE variables. This is an upper-bound internal-consistency task, not a deployment prediction task.
  - Task B: one-step-ahead dropped-connection prediction using current-row QoS/QoE variables while excluding current and future drop labels.
  - Task C: leakage-controlled one-step-ahead dropped-connection prediction using only session, scenario, device, application, and temporal context variables available before QoS outcomes.
- `ml_feature_sets.csv` records the exact feature columns used for each task.
- `ml_leakage_audit.csv` documents which columns were included, excluded, treated as targets, or used only as session-group identifiers.
- `ml_split_summary.csv` records the session-wise train/validation/test split structure for each task and random seed.
- `ml_benchmark_run_metrics.csv` stores run-level model performance across tasks, models, and random seeds.
- `ml_benchmark_summary.csv` summarizes mean and standard deviation values for precision, recall, F1-score, ROC-AUC, average precision, balanced accuracy, MCC, and Brier score.
- `ml_confusion_summary.csv` stores confusion-matrix counts for each task, model, and seed.
- `ml_feature_importance.csv` records feature-importance values for models that expose feature-importance estimates.
- `ml_reproducibility_metadata.json` records the dataset path, dataset SHA-256 hash, random seeds, split strategy, threshold-selection strategy, and benchmark interpretation.

The benchmark uses session-wise grouped splitting rather than row-wise random splitting, so rows from the same session are not shared across train, validation, and test sets. Thresholds are selected on the validation set, while the test set is used only once for reporting. Class imbalance is handled using class-weighted tree-based models where applicable, without SMOTE or row duplication.

The machine-learning results are interpreted as evidence of internal learnability under controlled synthetic benchmark conditions. Task A mainly checks whether the generated QoS/QoE structure is recoverable. Task B evaluates one-step-ahead prediction with current QoS/QoE variables. Task C is the most conservative benchmark because it uses only pre-QoS contextual variables.

The associated main benchmark figure is stored in `figures/ml_benchmark/ml_task_result_bars.pdf`. The precision-recall curve is treated as supplementary evidence and should be stored in `figures/supplementary/ml_precision_recall_curves.pdf`.

## Scope of the current external alignment

The external-alignment results provide evidence that selected SynNetQoS variables are comparable to external measurements in central tendency, especially for RSRP, download throughput, and controlled 5G jitter. The alignment should be interpreted as a distributional sanity check rather than as full empirical calibration of the generator.

The current workflow compares SynNetQoS against Vienna phone measurements and Campus QoS throughput data after mapping the available external variables into a common validation schema. RSRP and download throughput are used as the main external-alignment variables because they are directly represented in both SynNetQoS and the external datasets. Jitter is evaluated using the controlled 5G subset against Campus QoS data. Upload throughput is retained as secondary evidence because its alignment is less consistent across references.

## Scope of the current machine-learning benchmark

The machine-learning benchmark evaluates whether SynNetQoS contains structured, learnable relationships under leakage-aware splitting.

The benchmark is organized into three tasks. Task A is an upper-bound internal-consistency task because it predicts the same-timestep dropped-connection label using full current QoS/QoE variables. Task B predicts one-step-ahead dropped connection using current-row QoS/QoE variables while excluding the current and future drop labels. Task C is the most conservative task because it predicts one-step-ahead dropped connection using only contextual variables available before QoS outcomes.

The results show strong performance for the upper-bound internal-consistency task and lower but non-trivial performance for the leakage-controlled context-only task. This pattern is expected because the context-only task excludes direct QoS outcome variables and operates under class imbalance.

Accordingly, the benchmark supports the use of SynNetQoS for controlled synthetic-data evaluation and leakage-aware modelling experiments. Independent operational datasets would be required to assess real-world predictive deployment.

## Current limitations

Several limitations remain in the current validation setup.

First, the generator is not calibrated directly from real-world measurements. The external datasets are used for post-generation alignment checks, not for parameter fitting or optimization. Therefore, the results should not be interpreted as proof that SynNetQoS reproduces real deployment distributions.

Second, the comparison is limited by the external datasets available in the repository. The Vienna phone data provides large-scale RSRP and throughput measurements, but it does not provide latency or jitter variables for the same connected-user measurements. The Campus QoS data provides throughput and jitter, but the available sample size is smaller and does not include the same radio-context richness as the Vienna data.

Third, the current matching is performed at the variable and network-type level, not at a fully conditioned deployment level. The workflow does not yet match synthetic and external samples by operator, frequency band, base-station density, device model, precise mobility state, indoor/outdoor context, traffic load, or time-of-day. These missing conditioning variables can contribute to differences in spread and tail behavior between synthetic and external distributions.

Fourth, scanner measurements are treated only as supplementary RSRP evidence. Scanner data is useful for radio-signal context, but it is not equivalent to connected-phone QoS/QoE measurements and is therefore not mixed with throughput or jitter validation.

Fifth, packet-loss validation is not included because SynNetQoS does not currently expose a directly comparable packet-loss variable in the main generated dataset. Adding packet-loss generation would require explicit modeling assumptions and separate validation.

Sixth, some variables show stronger agreement in median values than in distributional spread. This is expected for a transparent synthetic generator that is designed around interpretable rules rather than fitted empirical distributions. The current outputs therefore support external alignment in selected variables, not exact real-world distribution matching.

Seventh, the machine-learning benchmark evaluates internal learnability on the generated dataset. Session-wise grouped splitting reduces leakage from repeated session rows, but external predictive validity would require independent operational datasets with comparable labels, features, and collection conditions.

## Future extension path

Future versions of SynNetQoS can strengthen empirical realism by adding a calibration stage that estimates generator parameters from external measurements. A stronger validation design could include:

- parameter calibration using real RSRP, throughput, latency, and jitter distributions;
- held-out external datasets reserved strictly for post-calibration evaluation;
- context-aware matching by network type, deployment condition, mobility state, indoor/outdoor status, and traffic load;
- explicit packet-loss generation and validation;
- operator- or region-specific generator profiles;
- uncertainty reporting across multiple external datasets;
- separate validation for LTE, 5G NSA, and 5G SA when the external metadata supports those distinctions;
- external machine-learning validation using independent real-world datasets with compatible dropped-connection or service-degradation labels.

These extensions would move SynNetQoS from an externally checked synthetic generator toward a more empirically calibrated network-measurement simulator.
