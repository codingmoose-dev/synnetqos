
# Data directory

This directory stores the data inputs used by the SynNetQoS repository.

The directory is intentionally separated into two parts:

1. `data/synthetic/` contains the SynNetQoS dataset generated by this repository.
2. `data/external/` contains independent reference datasets used for external alignment and validation checks.

This distinction is important. The synthetic dataset is the main generated research artifact of the project. The external datasets are not generated by SynNetQoS; they are used to compare selected QoS/QoE and radio-signal patterns against independent measurements.

```text
data/
├─ README.md
├─ external/
│  ├─ campus_qos/
│  │  ├─ ntnu_tput_all_Throughput.csv
│  │  └─ wue_tput_all_Throughput.csv
│  └─ vienna/
│     ├─ phone/
│     │  ├─ phone_data_5g.parquet
│     │  └─ phone_data_lte.parquet
│     └─ scanner/
│        ├─ scanner_data_5g.parquet
│        └─ scanner_data_lte.parquet
└─ synthetic/
   └─ synnetqos-dataset.csv
````

## Directory roles

| Directory                       | Role                                                     |
| ------------------------------- | -------------------------------------------------------- |
| `data/synthetic/`               | Stores the generated SynNetQoS dataset.                  |
| `data/external/campus_qos/`     | Stores controlled campus throughput reference data.      |
| `data/external/vienna/phone/`   | Stores phone-based LTE and 5G serving-cell measurements. |
| `data/external/vienna/scanner/` | Stores scanner-based LTE and 5G radio measurements.      |

## Synthetic dataset

The file `data/synthetic/synnetqos-dataset.csv` is the generated SynNetQoS dataset.

It contains session-level synthetic records with the following types of variables:

* temporal fields, such as timestamp, hour, day of week, and weekend indicator;
* user/session identifiers;
* deployment area and synthetic geographic coordinates;
* signal and network-capacity fields;
* offered and achieved uplink/downlink throughput;
* latency, jitter, and ping;
* network type, UE profile, operator profile, infrastructure profile, and band;
* mobility, congestion, weather, obstruction, indoor/outdoor, and tower-load context;
* QoE-related fields, including video-quality labels;
* event flags such as anomalous sessions and dropped connections.

The synthetic dataset is generated through the repository pipeline using the parameters defined in `config/generator_reference.yaml`. Derived summaries, integrity checks, and dataset hash information are written under `results/`.

The synthetic CSV should not be manually edited. Any correction should be made in the generator code or configuration, followed by regenerating the dataset.

## External reference datasets

The files under `data/external/` are independent reference datasets used for alignment checks. They are kept separate from the generated SynNetQoS dataset so that validation inputs are not confused with generated outputs.

### Campus QoS throughput data

The files in `data/external/campus_qos/` contain controlled throughput measurements.

Files included:

```text
data/external/campus_qos/ntnu_tput_all_Throughput.csv
data/external/campus_qos/wue_tput_all_Throughput.csv
```

The main fields include:

* actual uplink and downlink throughput;
* offered uplink and downlink throughput;
* uplink and downlink jitter;
* uplink and downlink loss;
* gNB implementation;
* SDR type;
* bandwidth;
* slot configuration;
* ratio;
* repetition index;
* scenario label.

These files are used for controlled throughput-oriented comparison against SynNetQoS outputs.

### Vienna phone measurements

The files in `data/external/vienna/phone/` contain phone-based LTE and 5G serving-cell measurements.

Files included:

```text
data/external/vienna/phone/phone_data_lte.parquet
data/external/vienna/phone/phone_data_5g.parquet
```

The phone files include radio-signal measurements and throughput-related fields. These are useful for comparing SynNetQoS signal and throughput distributions against phone-observed LTE and 5G measurements.

The LTE phone file includes fields such as:

* measurement ID and time;
* latitude and longitude;
* RSRP, RSRQ, RSSI, SINR, and path loss;
* downlink and uplink throughput;
* operator;
* timing advance;
* frequency, channel number, PCI, cell ID, eNB ID, and sector ID.

The 5G phone file includes similar fields, with additional beam-related fields such as beam index and beam type.

### Vienna scanner measurements

The files in `data/external/vienna/scanner/` contain scanner-based LTE and 5G radio measurements.

Files included:

```text
data/external/vienna/scanner/scanner_data_lte.parquet
data/external/vienna/scanner/scanner_data_5g.parquet
```

The scanner files are mainly used for signal-level comparison. They include radio and cell-identification fields but do not provide the same throughput fields as the phone datasets.

The scanner data are useful for checking whether the synthetic signal distributions remain within plausible ranges when compared with independent LTE and 5G radio measurements.

## How these files are used

The repository uses this directory as follows:

1. The generator creates the main SynNetQoS dataset under `data/synthetic/`.
2. The validation and external-alignment scripts read the generated dataset from `data/synthetic/`.
3. The same scripts read independent reference measurements from `data/external/`.
4. Numerical outputs are written under `results/`.
5. Figures are written under `figures/`.

The external datasets are used for comparison and alignment. They are not treated as generated SynNetQoS outputs.

## Data handling rules

Please follow these rules when working with this directory:

* Keep generated SynNetQoS data in `data/synthetic/`.
* Keep independent reference datasets in `data/external/`.
* Do not manually edit external reference files.
* Do not overwrite external files with cleaned or transformed versions.
* Save derived summaries, audits, and validation tables under `results/`.
* Save generated plots under `figures/`.
* Keep local path configuration in `config/paths.yaml`.
* Use `config/paths.example.yaml` as the template for expected external-data locations.

## Version-control note

The repository may track the generated SynNetQoS dataset and selected external reference files depending on file size, redistribution permissions, and project policy.

If an external dataset is not committed to the repository, the same folder structure should still be used locally. The validation scripts should then read the local paths through `config/paths.yaml`.

## Reproducibility note

The separation between `synthetic/`, `external/`, `results/`, and `figures/` is part of the repository design.

* `data/synthetic/` contains the generated dataset.
* `data/external/` contains independent reference inputs.
* `results/` contains derived numerical outputs.
* `figures/` contains plots generated from the analysis scripts.

This layout keeps the source data, generated data, validation inputs, and derived outputs traceable.
